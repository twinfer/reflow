No, you should **not** use IMStateMachine if you are storing your workflow states in Pebble DB.IMStateMachine stands for **In-Memory State Machine**. Instead of serializing memory into a file, Dragonboat expects you to utilize **Pebble's native snapshotting capabilities**.Here are the specific APIs you must implement inside your state machine to handle snapshots and log compaction:### 1. PrepareSnapshot() (The "Point-in-Time" Marker)Dragonboat periodically decides it needs to compact the Raft log to save disk space. Create a native Pebble snapshot.     // This is near-instant and doesn't stop incoming workflow processing.    snap := sm.db.NewSnapshot()        // 2. Return this snapshot handle. Dragonboat will pass this object     // straight into the SaveSnapshot method below.    return snap, nil}
```### 2. SaveSnapshot() (Streaming the Data)Immediately after preparing, Dragonboat invokes SaveSnapshot(). It hands you an io.Writer (which streams to a backup location or another node) and the snapshot handle you just created.```gofunc (sm *WorkflowStateMachine) SaveSnapshot(snapshotContext interface{}, w io.Writer, stopc <-chan struct{}) error {    snap := snapshotContext.(*pebble.Snapshot)    defer snap.Close()        // Iterate through all your workflow states inside this snapshot     // and write them to the io.Writer 'w'    iter := snap.NewIter(nil)    defer iter.Close()        for iter.First(); iter.Valid(); iter.Next() {        // Format your data into a binary stream (e.g., using gob, protobuf, or custom TLV)        if err := writeEntry(w, iter.Key(), iter.Value()); err != nil {            return err        }    }    return nil}
```Once SaveSnapshot returns successfully, Dragonboat knows it is safe to delete all the raw Raft logs on disk up to that snapshot index. **Your log compaction is complete.**### 3. RecoverFromSnapshot() (Node Recovery / Joint Cluster)If a node crashes and loses its disk, or if a brand new node joins your cluster, it won't have the history of logs to replay. Dragonboat will stream the snapshot file to that node and call RecoverFromSnapshot().```gofunc (sm *WorkflowStateMachine) RecoverFromSnapshot(r io.Reader, stopc <-chan struct{}) error {    // 1. Read the stream from 'r' and bulk-insert the keys back into Pebble    batch := sm.db.NewBatch()    for {        key, val, err := readEntry(r)        if err == io.EOF {            break        }        batch.Set(key, val, pebble.NoSync)    }    return batch.Commit(pebble.Sync)}
```## The Restate Constraint: The "Active Workflow" TrapWhile Dragonboat handles the *mechanics* of compaction via IOnDiskStateMachine, building a durable execution engine introduces a unique logical trap regarding snapshots.In a normal database, if a key is updated 1,000 times, the snapshot only needs to save the **final value**.However, in a durable execution engine like Restate, if a workflow invocation is **still running (suspended/waiting on a timer)**, you cannot afford to lose its historical execution steps. If you lose the log steps, you cannot replay the workflow to rebuild its execution context when the timer fires.### How to protect active histories:When writing your events to your Pebble DB inside Update(), store your workflow data under two types of keys: 1. **state:object_id** -> Stores the current state of a virtual object. (Safe to overwrite). 2. **journal:workflow_id:step_num** -> Stores historical side-effect execution results.When you write your SaveSnapshot logic, you must loop through your database and write *both* keys to the snapshot stream. However, you can write custom background logic inside your state machine to **garbage-collect (Delete)** all journal:workflow_id:* keys the moment a workflow officially hits a Completed or Failed status.This ensures that your Pebble snapshot files stay beautifully small and compacted, containing only active histories and final states.

Eliminate the "Double Network Hop" (The In-Process Worker Model)### The Shortfall:In Restate and Temporal, the architecture splits into a **Control Plane/Storage Engine** and an **Application Worker**. When a workflow executes a step (ctx.Run), it sends a gRPC/HTTP request across the network to the engine to register the step, waits for an acknowledgment, and then proceeds. The only delay is the Raft roundtrip to reach cluster consensus, drastically cutting latency.## 2. Introduce "Smart Execution Checkpoints" (Solving Replay Scaling)### The Shortfall:Both Temporal and Restate use **Replay-based execution**. If a workflow has 5,000 completed steps and crashes on step 5,001, the engine wakes up a new worker and re-executes the code from step 1, using the log history to instantly return the results of those first 5,000 steps. While elegant, this creates massive CPU spikes and severe memory consumption when long-running or large loop workflows crash.### The Right Design:Utilize your Pebble DB to introduce **Hybrid Execution Checkpoints** alongside your journal log. * Every time a step completes, it writes to the sequential Raft log (for cluster safety). * However, every X steps (or after an explicit structural loop), your engine serializes the actual execution state/variables and writes a single checkpoint row directly into your local Pebble DB. * If a crash occurs, instead of replaying thousands of entries from the beginning of time via GetLogReader, your engine queries Pebble for the latest checkpoint, loads the variable states instantly, and resumes *exactly* from step 4,900.## 3. Merge Pub/Sub Directly into the Execution Layer (The Routing Primitive)### The Shortfall:Durable execution platforms are fantastic for processing, but they are traditionally terrible at *routing communication*. If an external frontend client wants real-time progress updates from a running workflow, it cannot talk to the workflow directly. The worker has to write status updates to an external database, and the frontend client has to continuously poll that database.### The Right Design:Capitalize on Dragonboat's Multi-Raft layout by turning every Shard into a **Co-located Log + Pub/Sub Channel**. * When a client initiates a workflow, it opens a persistent WebSocket or gRPC stream directly to the engine node hosting that specific ShardID. * Because your engine knows exactly which node is the current Raft leader for that execution, it can stream live journal updates out of the state machine memory directly to the user's connection with zero database polling. The channel *is* the addressable primitive.## 4. Decouple "Active Journals" from "Object Snapshots" in Pebble### The Shortfall:Restate struggles with balancing state size. If an entity accumulates thousands of execution entries over months, the snapshot sizes grow, making partition split and node migration operations heavy and slow.### The Right Design:In your IOnDiskStateMachine implementation, segregate data in Pebble using a clean prefix schema that isolates volatile execution histories from long-term business states:```text├── [Prefix] state:obj_<id>      -> Pure application data (e.g., balance: $500)├── [Prefix] journal:wf_<id>     -> The active, uncompleted step histories└── [Prefix] index:active_wf     -> Index tracking which workflows are still alive
``` * **The Benefit:** When Dragonboat runs log compaction and triggers your SaveSnapshot() API, your code sweeps through Pebble, packing all the state:* entries, but **only streams the journal:* entries belonging to keys listed in index:active_wf**. * The moment a workflow completes, a background batch clears out its journal:* keys. You have just hit on what is universally considered **the hardest problem in durable execution.** In a standard web app, if you update code and redeploy, the old code vanishes and the new code takes over instantly.     ctx.Run(Step2)}
```### How it works under the hood: * When a brand new workflow starts, the engine logs "add-fraud-check": 1 into its execution journal inside Pebble. * For old workflows that were already running, the engine logs don't have that version key, so GetVersion returns 0. The old workflow safely skips the new step and completes using the old code path.## Solution 2: Sidecar Routing / Side-by-Side Deployments (The "Restate" Way)Instead of cluttering your Go code with if/else version checks, you treat each version of your workflow engine as an immutable deployment.If you have v1.0.0 of your binary running, and you want to release v1.1.0: 1. The v1.0.0 nodes keep their Dragonboat leadership and finish processing all the workflows that are currently in-flight. 3. Any **new** workflow execution requests are strictly routed by your API gateway to the v1.1.0 nodes. 4. Once the v1.0.0 nodes report that they have 0 active workflows left in their Pebble database, you safely shut them down.> **Why this rules for Embedded Engines:** Because Dragonboat supports **Online Cluster Membership changes**, you can seamlessly add the new v1.1.0 nodes into the Raft cluster, gracefully transfer leadership, and remove the old nodes without a single millisecond of downtime.> ## Solution 3: State Migration (The "Database" Way)Because your engine co-locates the workflow journal right next to your Pebble state store, you can do something neither Temporal nor Restate can easily do: **An active state migration.**If a workflow is suspended, its entire history is just rows in Pebble. ```You can trigger this migration by proposing a special SystemMigrationCommand through Dragonboat.
